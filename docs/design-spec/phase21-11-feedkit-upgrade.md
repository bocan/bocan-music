# Phase 21-11: FeedKit 9.1.2 to 10.4.0 upgrade

> Depends on: phases 21-1 through 21-10 complete (the whole Podcasts feature is
> in place and green). This slice retires the temporary FeedKit 9.1.2 pin called
> out in `phase21-notes.md` and moves the `Podcasts` module onto FeedKit
> **10.4.0** (latest stable, released 2026-02-05). It is a library-swap +
> refactor slice: no new user-facing surface, but it removes a Swift 6 concurrency
> workaround and unlocks `podcast:` namespace fields the old pin could not see.
>
> Read `_standards.md` and `phase21-0-overview.md` first; this slice keeps the
> `ParsedFeed` / `ParsedEpisode` boundary types unchanged in shape (plus one
> additive field), so nothing downstream of `FeedParser` has to change.
>
> This file resolves the "FeedKit version debt" and "Process note" sections of
> `phase21-notes.md`. New podcast functionality the upgrade makes possible (a
> transcript viewer, chapters, funding, OPML, and so on) is deliberately split
> out into `phase21-12-podcast-features.md`; this slice only does the upgrade and
> the "wire up everything FeedKit already parses" part of it.

## Goal

1. Replace the FeedKit `from: "9.1.2"` pin with `from: "10.4.0"`.
2. Delete the `@preconcurrency import FeedKit` workaround and the
   Sendable-boundary comments in `FeedParser.swift`. FeedKit 10.x types are
   `Sendable` (and `Codable` / `Equatable` / `Hashable`), so the escape hatch is
   no longer needed and `_standards.md` no longer tolerates it here.
3. Refactor `FeedParser.swift` to the 10.x API (nested RSS channel, de-prefixed
   iTunes fields, the new `Feed(data:)` throwing initializer, `FeedError`).
4. Wire up the podcast-relevant fields FeedKit 10.4.0 now exposes that 9.1.2
   could not: `podcast:transcript` (fills the currently-nil `transcript_url`),
   `podcast:guid` (a stable show identity), and Media RSS episode thumbnails (a
   better episode-artwork fallback). This is the "maximum support of FeedKit's
   podcast features" the brief asks for, bounded by what FeedKit actually parses.
5. Keep the blast radius tiny: only `FeedParser.swift`, the `Podcasts` manifest,
   `Package.resolved`, `NOTICES.md`, one small additive migration, and the
   Podcasts test fixtures change.

## Why now, and the stay-vs-replace decision

`phase21-notes.md` asks us to decide whether to stay on FeedKit or hand-roll an
`XMLParser` pipeline. **Decision: stay on FeedKit, upgrade to 10.4.0.** Rationale:

- FeedKit 10.4.0 is actively maintained (the 9.1.2 pin is roughly six years old;
  10.4.0 shipped 2026-02-05), built for Swift 6 with `Sendable` / `Codable`
  models, `swift-tools-version: 6.0`, and a macOS 12 floor (we target 15, so we
  are comfortably above it).
- It covers, in one parser, the things a hand-rolled solution would have to
  re-derive and keep correct against real-world feed sloppiness: RSS 2.0 + Atom +
  JSON Feed, RFC 822 and RFC 3339 date quirks, encoding detection, and the
  iTunes, Media RSS, Dublin Core, Content, Syndication, GeoRSS, and (partial)
  Podcast namespaces. Re-implementing that is high-risk for marginal benefit.
- It is a single MIT dependency with no system libraries.
- The one real gap (FeedKit parses only `podcast:guid` and `podcast:transcript`
  of the Podcasting 2.0 namespace, not `podcast:funding` / `podcast:chapters` /
  `podcast:person` / etc.) is small and can be filled with a localized
  supplementary parser later (see `phase21-12-podcast-features.md`) without
  abandoning FeedKit. That does not justify dropping the library.

So: upgrade, keep the thin normalization layer (`FeedParser` -> `ParsedFeed`),
and do not hand-roll.

## The 9.1.2 -> 10.4.0 API diff (the migration map)

Verify each row against the resolved 10.4.0 source before relying on it (see
"Context7 and source verification" below); the names here were read from FeedKit
`main` at the 10.4.0 line, but a leading-lowercase type name or an Atom title
shape is exactly the sort of thing worth re-confirming at implementation time.

### Packaging and entry point

| Concern | 9.1.2 (now) | 10.4.0 (target) |
|---------|-------------|-----------------|
| Pin | `.package(url: ".../FeedKit.git", from: "9.1.2")` | `from: "10.4.0"` |
| Sub-products | one `FeedKit` product | `FeedKit` plus an in-repo `XMLKit` product it depends on; SPM resolves `XMLKit` transitively, so the manifest still declares only `.product(name: "FeedKit", package: "FeedKit")` |
| Concurrency | classes, not `Sendable`; needs `@preconcurrency` | all models `Sendable` + `Codable` + `Equatable` + `Hashable`; no `@preconcurrency` |
| Parse call | `let r = FeedKit.FeedParser(data: data).parse()` returns `Result<Feed, ParserError>` | `let feed = try Feed(data: data)` (synchronous, throwing) |
| Error type | `ParserError` | `FeedError` |
| Feed enum | `.rss(RSSFeed)` / `.atom(AtomFeed)` / `.json(JSONFeed)` | same cases; also `feed.rss` / `feed.atom` / `feed.json` optional accessors |

Use the synchronous `Feed(data:) throws` initializer (the documented core
initializer). Do **not** switch to FeedKit's `urlString:` / `url:` async
initializers: those fetch the bytes themselves and would bypass `FeedFetcher`'s
conditional GET, size cap, `User-Agent`, redirect capture, and cancellation. The
fetch-then-parse split stays exactly as it is; only the parse line changes.

### RSS model: the channel is no longer flattened

In 9.x, channel fields hang directly off `RSSFeed`. In 10.x they live on
`RSSFeed.channel: RSSFeedChannel?`. Every `rss.X` becomes `rss.channel?.X`:

`rss.title` -> `rss.channel?.title`; `rss.description` -> `rss.channel?.description`;
`rss.link` -> `rss.channel?.link`; `rss.language` -> `rss.channel?.language`;
`rss.copyright` -> `rss.channel?.copyright`; `rss.managingEditor` ->
`rss.channel?.managingEditor`; `rss.image?.url` -> `rss.channel?.image?.url`;
`rss.items` -> `rss.channel?.items`; `rss.dublinCore?.dcCreator` ->
`rss.channel?.dublinCore?.creator` (note `dcCreator` -> `creator`); `rss.iTunes`
-> `rss.channel?.iTunes`; and the new `rss.channel?.podcast`.

### iTunes namespace: the `iTunes` prefix is gone

The `ITunes` umbrella type stays; its fields dropped the `iTunes` prefix. The
type that holds categories is now `iTunesCategory` (leading lowercase) and a
category's sub-entries are an array.

| 9.1.2 | 10.4.0 | Type note |
|-------|--------|-----------|
| `iTunes?.iTunesAuthor` | `iTunes?.author` | `String?` |
| `iTunes?.iTunesSummary` | `iTunes?.summary` | `String?` |
| `iTunes?.iTunesSubtitle` | `iTunes?.subtitle` | `String?` |
| `iTunes?.iTunesTitle` | `iTunes?.title` | `String?` |
| `iTunes?.iTunesImage?.attributes?.href` | `iTunes?.image?.attributes?.href` | unchanged shape |
| `iTunes?.iTunesExplicit` | `iTunes?.explicit` | still `String?` ("yes"/"no"/"explicit") |
| `iTunes?.iTunesDuration` | `iTunes?.duration` | `TimeInterval?` in both, already parsed |
| `iTunes?.iTunesSeason` | `iTunes?.season` | `Int?` |
| `iTunes?.iTunesEpisode` | `iTunes?.episode` | **`Int?` -> `String?`; convert with `Int($0)`** |
| `iTunes?.iTunesEpisodeType` | `iTunes?.episodeType` | `String?` |
| `iTunes?.iTunesOwner?.name` / `.email` | `iTunes?.owner?.name` / `.email` | unchanged |
| `iTunes?.iTunesCategories` (`[ITunesCategory]`) | `iTunes?.categories` (`[iTunesCategory]`) | type renamed |
| `category.subcategory?.attributes?.text` | `category.subcategory?.attributes?.text` | stays single (`iTunesSubCategory?`), NOT an array |
| also new | `iTunes?.type` | `String?`, "episodic" or "serial" |

### Item field access (RSS)

The container nests under `rss.channel?`, and two item fields were renamed:

- `item.enclosure?.attributes?.url` / `.type` / `.length` (unchanged shape;
  `RSSFeedEnclosure` is `XMLAttributesElement<RSSFeedEnclosureAttributes>` with
  `url: String?`, `type: String?`, `length: Int64?`).
- `item.guid?.text` (**changed**: `RSSFeedGUID` is now `XMLElement<...>`; the GUID
  value is `.text`, not `.value`).
- `item.content?.encoded` (**changed**: was `.contentEncoded`).
- `item.pubDate` (`Date?`), `item.title`, `item.link`, `item.description`
  (unchanged).

### Atom model

`AtomFeedTitle` / `AtomFeedSubtitle` / `AtomFeedContent` / `AtomFeedSummary` are
all `XMLElement<...>` typealiases, so their text value is `.text` (not `.value`).
The Atom entry title, however, is a plain `String?`.

| 9.1.2 | 10.4.0 |
|-------|--------|
| `atom.title` (`String?`) | `atom.title?.text` (title is `AtomFeedTitle?`) |
| `atom.subtitle?.value` | `atom.subtitle?.text` |
| `atom.logo` (`String?`) | unchanged |
| `atom.authors?.first?.name` / `.email` | unchanged |
| `atom.rights` | unchanged |
| `atom.links?...attributes?.rel` / `.href` | unchanged |
| `atom.entries` | unchanged |
| `entry.content?.value` / `entry.summary?.value` | `entry.content?.text` / `entry.summary?.text` |
| `entry.id` / `entry.published` / `entry.updated` | unchanged |
| `entry.title` | `String?` (a plain string, not the structured `AtomFeedTitle`) |

## Refactored `FeedParser.swift` (target shape)

Drop the `@preconcurrency` import and the boundary comment block. The public
signature is unchanged: `func parse(_ data: Data, sourceURL: URL) throws ->
ParsedFeed`. Sketch (illustrative; final field names per the verification step):

```swift
import Foundation
import FeedKit          // 10.4.0: models are Sendable; no @preconcurrency needed
import Observability

public struct FeedParser: Sendable {
    private let log = AppLogger.make(.podcasts)
    public init() {}

    public func parse(_ data: Data, sourceURL: URL) throws -> ParsedFeed {
        let feed: Feed
        do {
            feed = try Feed(data: data)
        } catch {
            // Unrecognized format vs malformed bytes both surface here; keep the
            // existing PodcastsError mapping so callers and tests are unaffected.
            throw PodcastsError.parseFailed(url: sourceURL, reason: String(describing: error))
        }

        switch feed {
        case let .rss(rss):   return try Self.parseRSS(rss, sourceURL: sourceURL)
        case let .atom(atom): return try Self.parseAtom(atom, sourceURL: sourceURL)
        case .json:           throw PodcastsError.notAFeed(url: sourceURL)
        }
    }
}
```

RSS channel mapping changes (showing only the deltas from today's code):

```swift
private static func parseRSS(_ rss: RSSFeed, sourceURL: URL) throws -> ParsedFeed {
    guard let channel = rss.channel,
          let title = channel.title, !title.isEmpty else {
        throw PodcastsError.notAFeed(url: sourceURL)
    }

    let author = channel.iTunes?.author
        ?? channel.managingEditor
        ?? channel.dublinCore?.creator

    let description = channel.iTunes?.summary ?? channel.description
    let artworkURL = (channel.iTunes?.image?.attributes?.href ?? channel.image?.url)
        .flatMap { URL(string: $0) }
    let categories = Self.parseITunesCategories(channel.iTunes?.categories)
    let podcastGUID = channel.podcast?.guid          // new: podcast:guid

    let episodes = (channel.items ?? []).compactMap { Self.parseRSSItem($0) }
    // ... sort newest-first as today, then:
    return ParsedFeed(
        title: title,
        author: author,
        description: description,
        artworkURL: artworkURL,
        link: channel.link.flatMap { URL(string: $0) },
        language: channel.language,
        explicit: Self.parseExplicit(channel.iTunes?.explicit),
        categories: categories,
        ownerName: channel.iTunes?.owner?.name,
        ownerEmail: channel.iTunes?.owner?.email,
        copyright: channel.copyright,
        fundingURL: nil,                  // still nil: FeedKit does not parse podcast:funding
        podcastGUID: podcastGUID,         // new
        episodes: sorted
    )
}
```

Item mapping deltas (episode number conversion, transcript, media thumbnail):

```swift
let episodeNumber = item.iTunes?.episode.flatMap { Int($0) }   // String? -> Int?

let title = item.title
    ?? item.iTunes?.title

return ParsedEpisode(

    duration: item.iTunes?.duration,        // TimeInterval?, unchanged
    season: item.iTunes?.season,
    episodeNumber: episodeNumber,
    episodeType: item.iTunes?.episodeType,
    artworkURL: artworkURL,
    chaptersURL: nil,                        // still nil: FeedKit does not parse podcast:chapters
```

Category helper, only the type name changes (`subcategory` is still single):

```swift
private static func parseITunesCategories(_ cats: [iTunesCategory]?) -> [String] {
    guard let cats else { return [] }
    var result: [String] = []
    for cat in cats {
        if let text = cat.attributes?.text { result.append(text) }
        if let sub = cat.subcategory?.attributes?.text { result.append(sub) }
    }
    return Array(Set(result)).sorted()
}
```

Transcript picker (single-column store; the full list is a 21-12 enhancement):

```swift
// Prefer a human-readable transcript URL. Store one URL in transcript_url; the
// full multi-language / multi-format list is deferred to phase 21-12.
private static func preferredTranscript(_ ts: [PodcastTranscript]?) -> URL? {
    guard let ts, !ts.isEmpty else { return nil }
    let ordered = ts.sorted { lhs, rhs in
        Self.transcriptRank(lhs.attributes?.type) < Self.transcriptRank(rhs.attributes?.type)
    }
    return ordered.lazy.compactMap { $0.attributes?.url.flatMap(URL.init(string:)) }.first
}
// Rank: text/vtt and application/json (timed) first, then srt, then plain, then unknown.
```

## Maximum FeedKit podcast support: what this slice does and does not fill

FeedKit 10.4.0's `podcast:` namespace models exactly two tags: `podcast:guid`
(channel-level, `channel.podcast?.guid`) and `podcast:transcript` (item-level,
`item.podcast?.transcripts`, each carrying `url` / `type` / `language` / `rel`).
There are no FeedKit models for `podcast:funding`, `podcast:chapters`,
`podcast:person`, `podcast:location`, `podcast:soundbite`, `podcast:value`, etc.

So "maximum support of FeedKit's podcast features" in this slice means:

**Newly wired up (was impossible on 9.1.2):**

- `transcript_url` <- `item.podcast?.transcripts` (best-effort single URL). This
  column existed since M023 but `FeedParser` always wrote `nil`.
- `podcast_guid` <- `channel.podcast?.guid` (new `ParsedFeed.podcastGUID` field +
  new `podcasts.podcast_guid` column via M024). The canonical cross-platform show
  identity; useful later for dedupe and Podcast Index correlation.
- Episode artwork fallback to `item.media?.thumbnails` (Media RSS) when
  `itunes:image` is absent. No new column: it just makes `artwork_url` better.
- `iTunes?.type` ("episodic" / "serial") is now readable. Persisting and using it
  (for a per-show default episode sort) is listed as a candidate in 21-12; this
  slice does not add a column for it, to keep M024 minimal.

**Still nil after this slice (FeedKit does not parse the tag):**

- `funding_url` (`podcast:funding`) and `chapters_url` (`podcast:chapters`)
  remain `nil`, exactly as today. Filling them requires a small supplementary
  parser, specced in `phase21-12-podcast-features.md`. Call this out in the PR so
  nobody assumes the upgrade alone lights up the funding button or chapters list.

## M024: additive `podcast_guid` column

`podcast:guid` has no home in the M023 `podcasts` table, so add one. The next
free migration number is **M024** (M023 is the highest registered; confirm at the
top of `Migrator.swift` before writing). Keep it additive and backfill-free:

```sql
ALTER TABLE podcasts ADD COLUMN podcast_guid TEXT;
```

Then add `podcastGUID: String?` to the `Podcast` record with `case podcastGUID =
"podcast_guid"`, include it in the `PodcastRepository` upsert column list, and map
it in `ParsedFeed.toPodcast(...)`. A refresh writes the value if the feed grows a
`podcast:guid`; existing rows stay `NULL` until their next refresh. Do not enforce
uniqueness on it (feeds can share or omit it); `feed_url` remains the identity.

If you would rather not touch Persistence in this slice, the alternative is to
add `ParsedFeed.podcastGUID` but not persist it yet, and move M024 into 21-12.
That leaves the parsed value unused (dead plumbing) in the interim, so the
recommendation is to land M024 here.

## Gotchas

- **The channel nesting is the easy thing to half-do.** Every RSS channel field
  moved under `rss.channel?`. Miss one and you read `nil` silently rather than
  failing to compile (most are optional already). Diff field-by-field against the
  table above.
- **`iTunes.episode` is a `String?` now.** A direct assignment compiles to the
  wrong type or, worse, if a field were `Any`-ish, smuggles a string where an
  `Int` is expected. Convert explicitly with `Int($0)`; feed it to both
  `ParsedEpisode.episodeNumber` and `fallbackTitle(episode:)`.
- **Text values moved from `.value` to `.text`.** The `XMLElement<...>`
  typealiases (`RSSFeedGUID`, `AtomFeedTitle`, `AtomFeedSubtitle`,
  `AtomFeedContent`, `AtomFeedSummary`) expose element text as `.text`. So
  `guid?.value` becomes `guid?.text`, and Atom `title`/`subtitle`/`content`/
  `summary` read via `.text`.
- **`content:encoded` is now `content?.encoded`** (was `.contentEncoded`).
- **`subcategory` stays single (`iTunesSubCategory?`), not an array.** Keep the
  single `cat.subcategory?.attributes?.text` access; only the category type name
  changed (`ITunesCategory` -> `iTunesCategory`).
- **Atom feed title is structured (`.text`) but the Atom entry title is a plain
  `String?`.** The asymmetry is real: `atom.title?.text` versus `entry.title`.
- **`explicit` is still a `String`**, not a `Bool`. Keep `parseExplicit(_:)`.
- **`duration` is already a `TimeInterval`** in both versions; do not re-add the
  string duration parsing the 21-2 gotchas warned about. FeedKit handles it.
- **Do not switch to the async `urlString:` initializers.** They would bypass
  `FeedFetcher`. Use `Feed(data:)` on the bytes you already fetched.
- **XMLKit is transitive, not a new top-level dependency.** Declare only the
  `FeedKit` product; SPM brings `XMLKit` in. One NOTICES note covers it.
- **The not-a-feed error case may shift** between `.parseFailed` and `.notAFeed`
  under 10.x (FeedKit may throw `unknownFeedFormat` where 9.x produced a
  titleless feed). The existing test asserts only `PodcastsError.self`, so it
  stays green; if you tighten it to a specific case, pick the case 10.x actually
  produces.

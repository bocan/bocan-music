# Phase 17 — Sane, Bullet-Proof CI/CD

> Prerequisites: v1.0.0 has shipped. The current pipeline (`ci.yml` + `release-please.yml` + `release.yml` + `website.yml` + the `homebrew-bocan` cross-repo dispatch) **must be replaced**, not extended.
>
> Read `phases/_standards.md` first. This phase replaces the entire `.github/workflows/` directory.

## Why this phase exists

The pipeline that shipped v0.x was assembled organically. It has these specific defects:

1. **Four interlocking workflows** with implicit ordering (`release-please` → manual PR merge → tag push → `release` → `workflow_run` → `website`) and no single source of truth for "what happens when I push a commit."
2. **Manual PR merging is required** for every release. There is no path from "I push a commit" to "users get an update" without human intervention beyond writing the commit message.
3. **CI commits back to `main`** (the appcast update). This is incompatible with branch protection. It also pollutes `git log` and creates loops that GitHub specifically blocks.
4. **Tags are managed by a third-party action** (release-please), not the developer. Re-running a release means deleting tags or using `workflow_dispatch` with manually-typed version strings.
5. **Cross-repo coupling via `repository_dispatch`** with an opaque JSON payload — no schema, no validation.
6. **Token sprawl** — `HOMEBREW_TAP_TOKEN`, `GITHUB_TOKEN`, the release-please bot, and several Apple credentials all overlap with no documented permission boundaries.
7. **No staging gate** between "DMG built" and "users notified" — a broken build goes straight to the appcast.
8. **`workflow_dispatch`-with-tag-input bypasses tag protection** — anyone with write access can publish an arbitrary version string.

## Goal

Replace the entire CI/CD surface with a single, comprehensible pipeline that satisfies these requirements:

- **Conventional Commits drive versioning automatically.** `fix:` → patch, `feat:` → minor, `feat!:` or `BREAKING CHANGE:` → major. The developer never types a version number.
- **One push = one release**, with no manual PR-merging step. If `main` advances and the new commits warrant a release, the release happens.
- **Zero CI commits to `main`.** All generated artifacts (appcast, DMG, sha256, release notes) live as GitHub release assets or on a dedicated branch (`gh-pages` for the website, `appcast` for the feed). `main` is for source code only.
- **Branch protection on `main` is enforced** — no direct pushes from anyone, including the maintainer. All changes go through a PR. CI must pass before merge.
- **A manual override exists** for: forcing a specific version bump, skipping a release, releasing from a non-`main` branch (hotfix tag), and re-running a release without retagging.
- **Every release is reproducible.** Given a tag, anyone with the secrets can rebuild the exact same DMG.
- **Cross-repo updates (Homebrew cask, future Sparkle channels) use a versioned event schema**, not free-form JSON.
- **Failures are loud and isolated.** A failed appcast update does not block the GitHub release. A failed cask update does not block the appcast.

## Non-goals

- Mac App Store distribution (still out of scope).
- Beta/RC channel automation beyond what Sparkle's `<sparkle:channel>` already provides. (The current `appcast-beta.xml` split survives, but no auto-routing of `*-beta` tags happens until needed.)
- Windows / Linux builds.
- Code signing key rotation tooling — manual for now.

## Outcome shape

```
.github/
├── workflows/
│   ├── ci.yml                  # PR + push validation. No release logic.
│   ├── release.yml             # Tag-driven build/sign/notarize/publish.
│   ├── tag.yml                 # Push-to-main → decide version → create tag (or skip).
│   ├── website.yml             # Deploys site on push to website/** OR on appcast update.
│   └── appcast.yml             # Updates appcast on release publish. Pushes to `appcast` branch.
├── actions/
│   ├── setup-bocan-build/      # Composite action: checkout + Xcode + brew + SPM cache + secrets stub.
│   ├── apple-sign/             # Composite action: keychain + import P12 + cleanup trap.
│   └── notarize-and-staple/    # Composite action: notarytool + stapler retry.
└── CODEOWNERS                  # Enforces review on every PR.

Scripts/
├── compute-next-version.sh     # Reads commits since last tag, returns major/minor/patch/none.
├── build-release.sh            # Single entry point — build + sign + notarize + DMG.
├── sparkle-update.sh           # (Existing) — generates one signed entry. No file mutation.
├── release-notes.sh            # (Existing) — extracts notes from CHANGELOG between tags.
└── verify-release.sh           # Smoke test: download DMG, mount, launch headless, verify version.

CHANGELOG.md                    # Generated by release.yml from commit history. Committed via PR, not directly.
```

The `release-please-config.json`, `.release-please-manifest.json`, and `release-please.yml` are **deleted**. release-please is replaced by `tag.yml` + a small bash script.

## Workflow responsibilities

### `ci.yml` — Validation

**Triggers:** every PR to `main`, every push to `main`.

**Jobs:**
- `lint` — SwiftLint, SwiftFormat, markdown lint, yamllint on `.github/workflows/`.
- `test` — `make test-coverage` + every SPM package's tests in parallel (matrix).
- `build-debug` — Debug archive only, no signing. Catches breakage that only shows up in archive builds.

**Outcome:** required for merge. Does **not** trigger any other workflow.

### `tag.yml` — Versioning

**Triggers:** push to `main` (post-merge of any PR).

**Behaviour:**
1. Read commits since last `v*.*.*` tag.
2. Run `Scripts/compute-next-version.sh` to classify them:
   - Any `feat!:` or `BREAKING CHANGE:` footer → major bump.
   - Any `feat:` → minor bump.
   - Any `fix:`, `perf:` → patch bump.
   - Only `chore:`, `docs:`, `style:`, `test:`, `build:`, `ci:`, `refactor:` → no release.
3. **Override mechanism:** if any commit message in the range contains:
   - `[release: major]` / `[release: minor]` / `[release: patch]` → forces that bump.
   - `[release: skip]` → exits without tagging even if commits would otherwise warrant a release.
   - `[release: x.y.z]` → forces an exact version (use sparingly — for re-publishing or hotfix renumbering).
4. If a release is warranted:
   - Update `CHANGELOG.md` and `Resources/Info.plist` via a **bot-authored PR** that auto-merges if CI passes (using `gh pr merge --auto --squash`).
   - The auto-merge of that PR triggers `tag.yml` again, but this run sees a `chore(release):` commit and exits at step 3.
   - **No, scratch that** — the simpler design is: `tag.yml` runs `git tag` directly (against the post-merge commit) and pushes the tag. CHANGELOG.md is generated and attached as a release asset, not committed. `Info.plist` is stamped at build time by `release.yml`, not pre-committed.

**Why this is better than release-please:** the maintainer never sees a release PR. There is no manual merge step. Versioning is deterministic from commit history.

**Override examples:**
```
fix: handle empty queue gracefully
[release: skip]
```
→ no release.
```
feat: add EQ presets

[release: major]
```
→ forces 2.0.0 even though `feat:` would normally bump minor.

### `release.yml` — Build, sign, publish

**Triggers:** `push: tags: ['v*.*.*']` only. **Remove** `release: types: [published]` and `workflow_dispatch` with arbitrary version input.

**Manual re-run:** to re-publish a tag, use the GitHub Actions "Re-run workflow" button on the failed run. To publish a hotfix from a non-`main` branch, push a tag from that branch — `release.yml` builds whatever commit the tag points at.

**Jobs (in order, each gated on the previous):**
1. `validate-tag` — Refuse to proceed if the tag is not on `main` or a `hotfix/*` branch. Refuse if the tag already has a published release (override with workflow input `force_republish: true`).
2. `build` — `Scripts/build-release.sh`. Outputs `Bocan.dmg`, `Bocan.dmg.sha256`, `appcast-entry.xml`. Uploads as artifacts.
3. `verify` — `Scripts/verify-release.sh`. Downloads the DMG artifact, mounts it, launches the app headlessly, asserts `CFBundleShortVersionString` matches the tag, asserts the app exits cleanly within 10 s. **This is the staging gate.**
4. `publish` — Create the GitHub release. Attach DMG, sha256, appcast entry, release notes.
5. `notify` — Fan out to two independent jobs that can fail without blocking each other:
   - `notify-appcast` — `repository_dispatch` to *this same repo* with `event-type: release-published`. Triggers `appcast.yml`.
   - `notify-cask` — `repository_dispatch` to `bocan/homebrew-bocan` with a **versioned schema** (see below).

### `appcast.yml` — Update the Sparkle feed

**Triggers:** `repository_dispatch: types: [release-published]` only.

**Behaviour:**
1. Check out the `appcast` branch (orphan branch, contains only `appcast.xml` and `appcast-beta.xml`).
2. Download the appcast entry from the release.
3. Prepend it to the appropriate feed.
4. Push to `appcast` branch.
5. Trigger `website.yml` via `workflow_run`.

**Why a separate branch:** the appcast updates many times per minor version. Keeping it off `main` means `git log main` is a clean history of code changes only. The website build pulls `appcast.xml` from the `appcast` branch as part of its build step.

### `website.yml` — Deploy

**Triggers:**
- `push` to `main` with paths `website/**`.
- `push` to `appcast` branch (any path).
- `workflow_dispatch`.

Build pulls `appcast.xml` and `appcast-beta.xml` from the `appcast` branch via `actions/checkout@v6` with `ref: appcast` into a subdirectory, then copies them into `website/_site/` before `actions/upload-pages-artifact`.

## Cross-repo event schema

Every `repository_dispatch` payload is versioned and validated. Schema at `.github/dispatch-schemas/release-published.v1.json`:

```json
{
  "schema_version": "1",
  "event_type": "release-published",
  "tag": "v1.2.3",
  "version": "1.2.3",
  "channel": "stable",
  "dmg_url": "https://github.com/bocan/bocan-music/releases/download/v1.2.3/Bocan.dmg",
  "dmg_sha256": "abc123...",
  "minimum_system_version": "26.0",
  "release_notes_url": "https://github.com/bocan/bocan-music/releases/tag/v1.2.3"
}
```

The receiving workflow in `homebrew-bocan` validates `schema_version == "1"` and rejects unknown event types. Any future field additions bump to `v2`.

## Branch protection

After this phase ships, configure on `main`:

- Require a pull request before merging.
- Require status checks: `lint`, `test`, `build-debug`.
- Require branches to be up to date.
- Require linear history (no merge commits).
- Require signed commits (developer + bot both use signed commits — bot uses GPG key stored as `BOT_GPG_PRIVATE_KEY` secret).
- Restrict who can push: empty list (no one, including admins).
- Allow force pushes: never.
- Allow deletions: never.

The `appcast` branch gets its own protection: only `appcast.yml` (via `GITHUB_TOKEN`) can push.

Tags `v*.*.*`: protected. Only `tag.yml` (via a fine-scoped PAT, `RELEASE_TAG_TOKEN`) can create them.

## Token inventory

After this phase, exactly these secrets exist with documented scope:

| Secret | Scope | Used by |
|---|---|---|
| `GITHUB_TOKEN` | Default per-job | All workflows for routine GitHub API |
| `RELEASE_TAG_TOKEN` | Fine-scoped PAT, `contents: write` on `bocan/bocan-music` only | `tag.yml` (creates protected tags) |
| `HOMEBREW_TAP_TOKEN` | Fine-scoped PAT, `contents: write` on `bocan/homebrew-bocan` only | `release.yml` notify-cask job |
| `BOT_GPG_PRIVATE_KEY` | GPG signing key for bot commits | `appcast.yml` |
| `DEVELOPER_ID_CERT_P12` + `DEVELOPER_ID_CERT_PASSWORD` | Apple Developer ID cert | `release.yml` build job |
| `DEVELOPER_ID_IDENTITY` | String of the identity name | `release.yml` build job |
| `APPLE_ID` + `APPLE_TEAM_ID` + `APP_SPECIFIC_PASSWORD` | Notarization credentials | `release.yml` build job |
| `SPARKLE_ED_PRIVATE_KEY` | EdDSA private key for appcast signing | `release.yml` build job |
| `ACOUSTID_API_KEY` + `BOCAN_LASTFM_API_KEY` + `BOCAN_LASTFM_SHARED_SECRET` | Third-party API keys baked into Release builds | `release.yml`, `ci.yml` |

No token has more permissions than it needs. Document this table in `SECURITY.md` as part of the phase.

## Migration plan

Order matters — do not skip steps.

1. **Land Phase 17 PRs in this order on `main` (still unprotected):**
   1. Add the new `Scripts/compute-next-version.sh` and `Scripts/verify-release.sh` with tests.
   2. Add the new composite actions under `.github/actions/`.
   3. Add `tag.yml`, `release.yml` (rewritten), `appcast.yml` (new), and updated `website.yml`. **Leave the old workflows in place but rename them to `*.yml.disabled`** so they don't run.
   4. Create the `appcast` orphan branch and seed it with the current `website/static/appcast.xml`.
   5. Update `homebrew-bocan` to validate the new schema. Bump its workflow to v2 atomically.
2. **Cut a test release** (`v1.0.1` or similar) end-to-end. Verify: tag created, DMG built, notarized, GitHub release published, appcast updated on `appcast` branch, website redeployed, cask PR opened.
3. **Delete** the old `*.yml.disabled` files, `release-please-config.json`, and `.release-please-manifest.json`.
4. **Enable branch protection on `main`** with the rules above.
5. **Enable branch protection on `appcast`** and tag protection on `v*.*.*`.

## Acceptance criteria

A reviewer (or `claude sonnet`, or any subagent) can verify this phase is complete by:

1. Running `gh workflow list` shows exactly: `CI`, `Tag`, `Release`, `Appcast`, `Website`. No release-please.
2. `git log main --author='github-actions' --oneline` shows zero commits after the migration date.
3. Pushing a `feat:` commit to a PR, merging it, and observing: tag created automatically, release built, DMG signed and notarized, appcast updated, website deployed, cask PR opened — all without manual intervention beyond merging the PR.
4. Pushing a `chore:` commit and observing no release happens.
5. Pushing `feat: foo\n\n[release: skip]` and observing no release happens.
6. Pushing `fix: bar\n\n[release: minor]` and observing a minor bump (override worked).
7. Attempting `git push origin main` directly fails with "protected branch" error.
8. Re-running a failed `release.yml` run from the GitHub UI succeeds without retagging.
9. The DMG produced by the pipeline contains exactly one item (`Bocan.app`) — no `DistributionSummary.plist`, `ExportOptions.plist`, or `Packaging.log`.
10. `brew livecheck bocan` returns the latest version within 5 minutes of release.

## Things explicitly **not** to do in this phase

- Do not introduce additional channels (beta/nightly) until users ask. The infrastructure supports it via `[release: prerelease]` flag and Sparkle's `<sparkle:channel>`, but wiring up auto-routing now is YAGNI.
- Do not move to a release-train model (scheduled releases on a cadence). Continuous delivery driven by commit semantics is simpler.
- Do not introduce a release dashboard, deployment service, or observability stack. The GitHub Actions UI is sufficient.
- Do not migrate to GitLab/Forgejo/anything else as part of this phase. CI/CD redesign and platform migration are separate concerns.
- Do not ship Sparkle "delta updates" yet. Full DMG every time. Bandwidth is cheap, complexity is not.

## Open questions for the maintainer to decide before starting

1. Does `compute-next-version.sh` look at **squash-merge commit messages** (which inherit the PR title) or **all commits in the range** (which includes the original feature-branch commits)? Recommendation: squash-merge messages only, since the maintainer controls the squash message at merge time. This makes overrides trivially auditable.
2. Should the bot use a real GitHub App identity (more setup, cleaner audit trail) or a fine-scoped PAT (faster to set up, less ideal long-term)? Recommendation: PAT for v1.x, GitHub App once the project has more contributors.
3. Should hotfix branches (`hotfix/*`) be allowed to publish releases without a PR? Recommendation: yes for `hotfix/*` only, with an audit-log notification.
